# UNIX 标准 I/O 库：从 `FILE *` 到内核文件描述符

## 1. 本章到底要解决什么问题

《UNIX 环境高级编程》第三章已经介绍了文件 I/O：

```
open()
read()
write()
lseek()
close()
```

这些函数直接操作内核提供的**文件描述符**，特点是：

- 操作对象是 `int fd`
- 读写单位是原始字节
- 每次 `read()`、`write()` 都可能发生系统调用
- 缓冲区、分行、格式转换、错误状态需要程序自己处理

第五章标准 I/O 库则在文件描述符之上增加了一层用户态封装：

```
fopen()
fgets()
fread()
fprintf()
fflush()
fclose()
```

它主要解决三个问题：

1. **通过用户态缓冲减少系统调用次数**
2. **提供按字符、按行、按块、按格式的高级读写接口**
3. **通过** `**FILE \***` **统一维护缓冲区、文件位置和错误状态**

因此，本章的核心并不是单纯学习一组新 API，而是理解下面这套分层模型：

```
业务代码
   ↓
FILE * 标准 I/O 流
   ↓
用户态缓冲区
   ↓
read / write
   ↓
内核文件描述符
   ↓
文件、终端、管道
```

# 2. 标准 I/O 的整体知识结构

## 2.1 第一层：业务代码

业务代码通常不会关心底层每次究竟读取了多少字节，而是希望使用更自然的方式处理数据。

例如按行读取配置文件：

```
char line[1024];

while (fgets(line, sizeof(line), fp) != NULL) {
    parse_config_line(line);
}
```

或者格式化写入日志：

```
fprintf(fp, "user_id=%d, status=%s\n", user_id, status);
```

业务代码表达的是：

```
读取一行
写入一条格式化记录
读取一个结构体数组
输出一个字符串
```

而不是：

```
调用 read() 读取 4096 字节
自己搜索换行符
处理半行数据
保存剩余数据
下次继续拼接
```

标准 I/O 库的作用，就是把这些低层细节隐藏在 `FILE *` 内部。

## 2.2 第二层：`FILE *` 标准 I/O 流

`FILE *` 不是文件本身，也不是单纯的文件描述符。

它是 C 标准库维护的一个**流对象**，内部通常包含：

```
底层文件描述符
缓冲区地址
缓冲区容量
当前读写位置
缓冲区中有效数据的范围
EOF 标志
错误标志
读写模式
缓冲模式
```

可以用下面的简化结构理解：

```
struct FILE {
    int fd;

    char *buffer;
    size_t capacity;

    size_t current;
    size_t available;

    bool eof;
    bool error;
};
```

真实的 libc 实现会复杂得多，但这个模型足以帮助我们理解标准 I/O。

因此：

```
文件描述符 fd
```

表示的是：

> 内核中的一个打开文件对象。

而：

```
FILE *
```

表示的是：

> 标准库在文件描述符之上维护的用户态流对象。

二者的关系大致是：

```
FILE *
 ├── 底层 fd
 ├── 用户态缓冲区
 ├── 当前缓冲位置
 ├── EOF 状态
 ├── 错误状态
 └── 读写模式
```

## 2.3 第三层：用户态缓冲区

用户态缓冲区是标准 I/O 库最核心的机制。

如果没有缓冲区，业务代码每次调用：

```
fgetc(fp);
```

都可能对应一次：

```
read(fd, &ch, 1);
```

假设读取一个 1 MB 文件，就可能产生上百万次系统调用，性能会非常差。

标准 I/O 不会这样工作。它通常会一次从内核读取一批数据：

```
read(fd, buffer, 4096);
```

然后让多次 `fgetc()`、`fgets()` 从用户态缓冲区中消费这些数据。

因此，标准 I/O 的性能收益主要来自：

```
少量的大块系统调用
代替
大量的小块系统调用
```

缓冲区本身并不会让磁盘或网络变快，它只是减少了：

- 用户态与内核态切换
- 系统调用次数
- 小块 I/O 带来的额外开销

## 2.4 第四层：`read()` / `write()`

标准 I/O 库最终仍然需要通过系统调用访问内核。

例如：

```
fgets(line, sizeof(line), fp);
```

底层可能触发：

```
read(fd, internal_buffer, buffer_size);
```

例如：

```
fprintf(fp, "hello\n");
```

在刷新缓冲区时，底层可能触发：

```
write(fd, internal_buffer, data_size);
```

因此：

```
fgets / fread / fprintf / fwrite
```

本身不是内核系统调用，而是用户态库函数。

真正进入内核的是：

```
read / write
```

这也是标准 I/O 与文件 I/O 的本质关系：

```
标准 I/O 是对文件 I/O 的用户态封装
```

## 2.5 第五层：内核文件描述符

当调用：

```
FILE *fp = fopen("data.txt", "r");
```

标准库内部通常会：

1. 根据 `"r"` 转换出对应的打开标志
2. 调用 `open()`
3. 获取文件描述符
4. 创建并初始化一个 `FILE` 流对象
5. 把文件描述符保存到 `FILE` 内部

逻辑上近似于：

```
int fd = open("data.txt", O_RDONLY);
FILE *fp = create_file_stream(fd);
```

同样地：

```
fclose(fp);
```

通常会完成：

1. 刷新待写数据
2. 调用 `close(fd)`
3. 释放 `FILE` 对象
4. 释放用户态缓冲区

所以 `fclose()` 不只是简单调用 `close()`。

## 2.6 第六层：文件、终端和管道

文件描述符并不只代表磁盘文件。

它还可以代表：

- 普通文件
- 终端
- 管道
- FIFO
- socket
- 设备文件

因此标准 I/O 流也可以建立在不同类型的文件描述符之上。

例如程序启动时通常已经存在三个标准流：

```
stdin
stdout
stderr
```

它们分别对应：

```
stdin  → STDIN_FILENO  → fd 0
stdout → STDOUT_FILENO → fd 1
stderr → STDERR_FILENO → fd 2
```

但具体连接到什么对象，取决于程序的运行环境。

直接运行：

```
./server
```

可能是：

```
stdout → 终端
```

重定向运行：

```
./server > server.log
```

则变成：

```
stdout → 普通文件
```

通过管道运行：

```
./server | grep error
```

则变成：

```
stdout → 管道
```

业务代码仍然调用：

```
printf()
```

但底层目标已经不同。

这就是 UNIX “一切皆文件”思想在标准 I/O 中的体现。

# 3. 标准 I/O 的读取流程

考虑下面的代码：

```
FILE *fp = fopen("data.txt", "r");

char line[128];
fgets(line, sizeof(line), fp);
```

## 3.1 第一次读取

第一次调用 `fgets()` 时，`FILE` 内部缓冲区通常是空的。

此时完整流程如下：

```
业务代码调用 fgets()
        ↓
检查 FILE 内部缓冲区
        ↓
缓冲区为空
        ↓
调用 read(fd, internal_buffer, N)
        ↓
内核从文件中读取一批数据
        ↓
数据进入 FILE 的用户态缓冲区
        ↓
fgets() 在缓冲区中查找换行符
        ↓
只返回一行给业务代码
        ↓
剩余数据继续留在缓冲区
```

假设文件内容为：

```
hello
world
unix
```

底层一次 `read()` 可能已经读取：

```
hello\nworld\nunix\n
```

但第一次 `fgets()` 只返回：

```
hello\n
```

缓冲区中的剩余内容为：

```
world\nunix\n
```

## 3.2 第二次读取

业务代码再次调用：

```
fgets(line, sizeof(line), fp);
```

此时标准库发现缓冲区中仍然存在数据，因此：

```
不会立即调用 read()
```

而是直接从缓冲区返回：

```
world\n
```

只有当缓冲区的数据全部消费完毕后，标准库才会再次调用 `read()`。

## 3.3 读取路径总结

标准 I/O 的读取路径可以概括为：

```
文件、终端、管道
        ↓
内核文件描述符
        ↓ read()
FILE 用户态缓冲区
        ↓ fgets/fgetc/fread/fscanf
业务代码
```

需要重点理解：

> 一次标准 I/O 读取，不一定对应一次系统调用。

也就是说：

```
fgets()
```

与：

```
read()
```

不是一一对应关系。

标准库可能：

- 一次 `read()` 支撑多次 `fgets()`
- 一次 `read()` 支撑大量 `fgetc()`
- 在一次 `fread()` 中执行多次 `read()`

具体行为取决于：

- 缓冲区大小
- 请求的数据大小
- 底层对象类型
- libc 实现
- 是否遇到 EOF 或错误

# 4. 标准 I/O 的写入流程

考虑下面的代码：

```
FILE *fp = fopen("server.log", "w");

fprintf(fp, "request_id=%d\n", request_id);
```

写入流程一般为：

```
业务代码调用 fprintf()
        ↓
格式化整数、字符串等数据
        ↓
把格式化后的字符写入 FILE 用户态缓冲区
        ↓
缓冲区未满，暂时不调用 write()
        ↓
后续继续积累数据
        ↓
满足刷新条件
        ↓
调用 write(fd, buffer, size)
        ↓
数据进入内核
```

因此：

```
fprintf(fp, "hello");
```

返回时，数据可能仍然只存在于用户态缓冲区中。

它不一定已经：

- 进入内核
- 写入文件系统
- 写入磁盘

## 4.1 写入路径总结

```
业务代码
   ↓ fprintf/fputs/fwrite
FILE 用户态缓冲区
   ↓ write()
内核文件描述符
   ↓
内核页缓存、管道、终端或设备
```

标准 I/O 的写缓冲同样是为了：

```
把多次小规模写入合并成少量大规模 write()
```

例如业务代码连续执行：

```
fprintf(fp, "name=%s ", name);
fprintf(fp, "age=%d ", age);
fprintf(fp, "status=%s\n", status);
```

标准库可能并不会产生三次 `write()`。

它可能先把这些数据全部放入用户态缓冲区，最后只执行一次：

```
write(fd, buffer, total_size);
```

# 5. 三种缓冲模式

标准 I/O 通常存在三种缓冲方式。

## 5.1 全缓冲

普通文件通常使用全缓冲。

特点是：

```
缓冲区满时才自动刷新
```

例如：

```
FILE *fp = fopen("output.txt", "w");
fprintf(fp, "hello");
```

此时 `hello` 可能仍在用户态缓冲区中。

全缓冲适合普通文件，因为普通文件通常更关注吞吐量，而不是每条数据的即时可见性。

典型行为：

```
多次 fprintf/fwrite
        ↓
用户态缓冲区逐渐填满
        ↓
缓冲区满
        ↓
调用 write()
```

## 5.2 行缓冲

连接到终端的标准输出通常采用行缓冲。

行缓冲常见的刷新条件是遇到换行符：

```
printf("hello\n");
```

通常会及时显示。

但下面的代码：

```
printf("hello");
```

可能不会立即显示，因为：

- 没有换行
- 缓冲区没有满
- 没有主动调用 `fflush(stdout)`

因此交互式命令行程序常见写法是：

```
printf("请输入用户名：");
fflush(stdout);

scanf("%s", name);
```

否则提示信息可能仍在缓冲区中，用户却已经被程序阻塞等待输入。

需要注意：

> 行缓冲不等于每次调用输出函数都会刷新。

## 5.3 不缓冲

不缓冲意味着数据尽可能直接交给底层文件描述符。

标准错误流 `stderr` 通常不缓冲或具有更强的即时输出语义。

例如：

```
fprintf(stderr, "fatal error\n");
```

错误输出需要尽快可见，因为程序可能马上崩溃或退出。

不过，不缓冲并不意味着没有任何内核缓冲。

它只是意味着：

```
标准 I/O 库不在用户态长时间积累数据
```

数据经过 `write()` 后，仍然可能进入：

- 内核页缓存
- 终端驱动缓冲区
- 管道缓冲区
- socket 发送缓冲区

# 6. `fflush()`、`fclose()` 与真正落盘

这是本章最重要、也最容易混淆的知识点之一。

## 6.1 `fflush()`

```
fflush(fp);
```

作用是：

> 把标准 I/O 用户态输出缓冲区中的数据交给底层文件描述符。

也就是：

```
FILE 用户态缓冲区
        ↓ fflush()
write()
        ↓
内核
```

`fflush()` 成功并不意味着数据已经写入物理磁盘。

它通常只能保证：

```
数据已经从 C 标准库的缓冲区进入内核
```

## 6.2 `fsync()`

如果需要要求内核把文件数据同步到底层存储设备，可以使用：

```
fflush(fp);

int fd = fileno(fp);
fsync(fd);
```

完整路径为：

```
业务代码
   ↓
FILE 用户态缓冲区
   ↓ fflush()
内核页缓存
   ↓ fsync()
磁盘或持久化设备
```

因此：

```
fflush()
```

解决的是用户态缓冲刷新。

而：

```
fsync()
```

解决的是内核缓存向持久化设备同步。

二者不是同一个层次。

## 6.3 为什么不能频繁调用 `fsync()`

`fsync()` 可能需要等待实际设备完成写入，成本远高于普通的 `write()`。

如果每写一条日志都执行：

```
fprintf(fp, ...);
fflush(fp);
fsync(fileno(fp));
```

程序吞吐量可能大幅下降。

生产环境中需要根据业务语义做取舍：

| 场景         | 常见策略                     |
| ------------ | ---------------------------- |
| 普通调试日志 | 异步批量写入                 |
| 普通访问日志 | 定期刷新                     |
| 核心交易记录 | 根据持久化要求执行同步       |
| 崩溃诊断信息 | 尽快输出到 `stderr`          |
| 数据库 WAL   | 由数据库根据事务策略控制同步 |

不能把“每次立即落盘”当成通用最佳实践。

## 6.4 `fclose()`

```
fclose(fp);
```

通常需要执行：

```
刷新标准 I/O 输出缓冲区
        ↓
调用 close() 关闭文件描述符
        ↓
释放 FILE 对象
        ↓
释放用户态缓冲区
```

因此，写文件时应当检查 `fclose()` 的返回值：

```
if (fclose(fp) == EOF) {
    perror("fclose");
}
```

某些写入错误可能不会在 `fprintf()` 时立即暴露，而是在缓冲区真正刷新时才发现。

例如：

- 磁盘空间不足
- 文件系统错误
- 网络文件系统异常
- 管道另一端已经关闭

# 7. 标准 I/O 的常见接口分类

不需要死记所有 API，应该按照功能建立分类。

## 7.1 打开和关闭流

```
fopen()
freopen()
fdopen()
fclose()
```

### `fopen()`

根据路径创建标准 I/O 流：

```
FILE *fp = fopen("config.txt", "r");
```

常见模式：

| 模式   | 含义                   |
| ------ | ---------------------- |
| `"r"`  | 只读，文件必须存在     |
| `"w"`  | 只写，清空原文件或新建 |
| `"a"`  | 追加写                 |
| `"r+"` | 可读可写，文件必须存在 |
| `"w+"` | 可读可写，清空或新建   |
| `"a+"` | 可读并追加写           |

### `fdopen()`

把已有文件描述符包装成 `FILE *`：

```
int fd = open("data.txt", O_RDONLY);
FILE *fp = fdopen(fd, "r");
```

适用于：

- 已经通过 `open()` 获得文件描述符
- 第三方库返回文件描述符
- 管道文件描述符
- 某些需要标准 I/O 高层接口的场景

一旦交给 `fdopen()` 管理，通常应由：

```
fclose(fp);
```

负责关闭底层文件描述符，而不是再单独调用 `close(fd)`。

### `fileno()`

获取 `FILE *` 内部的底层文件描述符：

```
int fd = fileno(fp);
```

常见用途：

- 调用 `fsync()`
- 调用 `fstat()`
- 设置文件描述符属性
- 与某些只接受 `fd` 的系统接口协作

## 7.2 按字符读写

```
fgetc()
getc()
getchar()

fputc()
putc()
putchar()
```

适合：

- 简单字符解析
- 小型文本处理
- 教学或工具程序

在性能敏感业务中，不应频繁逐字符处理大型文件，除非标准库缓冲已经足够且解析逻辑确实需要逐字符状态机。

## 7.3 按行读写

```
fgets()
fputs()
```

读取文本文件时通常优先使用：

```
char line[1024];

while (fgets(line, sizeof(line), fp) != NULL) {
    // 处理 line
}
```

与不安全的 `gets()` 不同，`fgets()` 会接收缓冲区大小，可以避免直接越界。

需要注意：

> 如果一行长度超过缓冲区大小，一行数据可能会被拆成多次读取。

因此不能默认每次 `fgets()` 都一定获得完整的一行。

生产代码应根据输入格式决定：

- 限制单行最大长度
- 检测是否读取到换行符
- 动态扩容并继续拼接
- 使用 `getline()`

## 7.4 按块读写

```
fread()
fwrite()
```

适合：

- 二进制文件
- 批量数据读写
- 固定大小记录
- 离线数据处理

例如：

```
char buffer[4096];

size_t n;
while ((n = fread(buffer, 1, sizeof(buffer), fp)) > 0) {
    process(buffer, n);
}
```

写入：

```
size_t n = fwrite(buffer, 1, data_size, fp);

if (n != data_size) {
    // 处理写入失败
}
```

需要注意返回值表示的是成功处理的“元素个数”，而不是必然表示字节数：

```
fread(ptr, element_size, element_count, fp);
```

返回值范围为：

```
0 到 element_count
```

## 7.5 格式化 I/O

```
printf()
fprintf()
sprintf()
snprintf()

scanf()
fscanf()
sscanf()
```

格式化输出非常方便：

```
fprintf(fp,
        "request_id=%llu cost_ms=%d status=%s\n",
        request_id,
        cost_ms,
        status);
```

但是格式化输入需要谨慎。

例如：

```
fscanf(fp, "%s", buffer);
```

如果没有限制长度，可能导致缓冲区溢出。

相比 `sprintf()`，更推荐使用：

```
snprintf(buffer, sizeof(buffer), ...);
```

因为它可以限制最大写入长度。

在现代 C++ 服务端项目中，格式化通常会进一步使用：

- `std::format`
- `fmt`
- 日志库提供的格式化接口

但它们解决的问题与 `fprintf()` 类似：

```
把类型化数据转换成文本表示
```

# 8. EOF 和错误处理

## 8.1 EOF 不是文件中的特殊字符

`EOF` 是标准库定义的一个特殊返回值，用于表示：

```
没有成功读取下一个字符
```

它可能由两种原因造成：

1. 正常到达文件末尾
2. 读取过程中发生错误

例如：

```
int ch;

while ((ch = fgetc(fp)) != EOF) {
    process(ch);
}
```

循环结束后需要进一步判断：

```
if (ferror(fp)) {
    perror("read failed");
} else if (feof(fp)) {
    // 正常到达文件末尾
}
```

不能简单认为：

```
返回 EOF = 正常读完文件
```

## 8.2 `feof()` 的正确理解

`feof(fp)` 并不是用来预测下一次读取是否会失败。

错误写法：

```
while (!feof(fp)) {
    fgets(line, sizeof(line), fp);
    process(line);
}
```

因为 EOF 标志通常是在一次读取操作失败后才设置。

正确写法是：

```
while (fgets(line, sizeof(line), fp) != NULL) {
    process(line);
}

if (ferror(fp)) {
    // 读取错误
}
```

核心原则：

> 应当以读取函数的返回值控制循环，而不是提前检查 `feof()`。

## 8.3 清除流状态

可以使用：

```
clearerr(fp);
```

清除：

- EOF 标志
- 错误标志

但清除标志并不能修复底层错误。

它只是允许程序在已经处理问题后继续使用该流。

# 9. 文件位置与缓冲区的关系

标准 I/O 流内部存在用户态缓冲，因此业务代码看到的逻辑位置，与底层文件描述符当前的内核偏移量可能并不完全一致。

例如：

```
fgets(line, sizeof(line), fp);
```

业务代码只获得了一行。

但是标准库底层可能已经通过一次 `read()` 提前读取了几 KB 数据。

此时：

```
业务代码认为：只读取了一行
FILE 缓冲区：保存了后续多行
底层 fd 偏移量：已经向后移动了几 KB
```

因此，标准 I/O 需要使用自己的定位接口：

```
ftell()
fseek()
rewind()
fgetpos()
fsetpos()
```

而不要随意绕过 `FILE *`，直接对底层文件描述符调用：

```
lseek()
```

否则可能导致：

- 缓冲区中的数据失效
- 文件位置不一致
- 重复读取
- 跳过部分数据
- 行为难以推断

# 10. 为什么不应随意混用 `FILE *` 和 `fd`

下面的代码存在风险：

```
FILE *fp = fopen("data.txt", "r");
int fd = fileno(fp);

char line[128];
fgets(line, sizeof(line), fp);

char buffer[128];
read(fd, buffer, sizeof(buffer));
```

问题在于：

```
fgets()
```

可能已经提前从 `fd` 中读取了大量数据，保存在 `FILE` 的用户态缓冲区中。

随后直接调用：

```
read(fd, ...)
```

读取的位置是内核文件描述符当前的位置，而不是业务代码认为的“下一行”。

结果可能跳过 `FILE` 缓冲区中尚未消费的数据。

可以把状态理解为：

```
文件内容：
A B C D E F G H

FILE 第一次预读：
A B C D E F

业务代码只消费：
A B

FILE 缓冲区剩余：
C D E F

底层 fd 已经指向：
G

此时直接 read(fd)：
从 G 开始读取
```

于是 `C D E F` 被绕过了。

工程原则：

> 同一个打开文件应明确选择一种主要访问模型：要么使用标准 I/O，要么使用文件描述符 I/O。

确实需要混用时，必须明确处理：

- 缓冲区刷新
- 文件位置同步
- 读写方向切换
- 错误状态
- 所有权和关闭责任

# 11. 读写流切换的注意事项

对于以更新模式打开的流：

```
FILE *fp = fopen("data.txt", "r+");
```

同一个流既可以读，也可以写。

但不能毫无同步地在读写操作之间反复切换。

通常应当在方向切换时使用：

```
fflush()
```

或者文件定位操作：

```
fseek()
fsetpos()
rewind()
```

例如：

```
fread(buffer, 1, size, fp);

fseek(fp, 0, SEEK_CUR);

fwrite(data, 1, data_size, fp);
```

根本原因仍然是：

```
读缓冲区和写缓冲区都属于 FILE 的内部状态
```

方向切换必须让标准库有机会同步这些状态。

# 12. 标准 I/O 在 C++ 服务端中的位置

## 12.1 不适合直接作为高并发网络 I/O 核心

高并发网络服务通常不会把 socket 简单包装为 `FILE *`，然后使用：

```
fgets()
fprintf()
```

核心原因是高并发服务需要精确控制：

- 非阻塞 I/O
- 部分读取和部分写入
- ET/LT 触发模式
- 事件循环
- 应用层缓冲区
- 消息边界
- 背压
- 超时
- 连接生命周期

服务端核心网络层通常使用：

```
socket
recv / send
read / write
readv / writev
epoll
io_uring
```

并自行维护：

```
输入缓冲区
输出缓冲区
协议解析状态
连接状态
```

例如：

```
socket 可读
   ↓
recv() 写入连接输入缓冲区
   ↓
协议解析器判断是否收到完整请求
   ↓
生成业务任务
   ↓
业务线程处理
   ↓
响应写入连接输出缓冲区
   ↓
send() 逐步发送
```

这里不能依赖标准 I/O 内部不可见的预读和缓冲行为。

## 12.2 标准 I/O 仍然适用的场景

标准 I/O 在下面这些场景中仍然非常常见：

### 读取简单配置文件

```
while (fgets(line, sizeof(line), fp)) {
    parse_config(line);
}
```

### 编写命令行工具

例如：

- 日志分析
- 文本过滤
- 文件转换
- 批量处理脚本配套程序

### 离线数据处理

例如：

- 顺序扫描大文件
- 读取训练数据
- 导出统计结果

### 调用传统 C 库

很多 C 库接口直接使用：

```
FILE *
```

### 简单文本和二进制文件处理

当业务不需要异步、非阻塞和精确缓冲控制时，标准 I/O 可以显著减少代码量。

## 12.3 日志系统通常不会直接依赖 `fprintf()`

简单程序可以：

```
fprintf(log_fp, "request failed\n");
```

但生产级日志系统通常还需要：

- 多线程安全
- 日志级别
- 时间戳
- 日志滚动
- 异步写入
- 批量刷新
- 崩溃处理
- 格式化
- 丢弃策略
- 磁盘空间保护

典型结构是：

```
业务线程
   ↓
格式化日志记录
   ↓
无锁队列或受控队列
   ↓
日志线程
   ↓
批量 write()
   ↓
日志文件
```

不过，这些日志系统仍然是在解决与标准 I/O 相同的核心问题：

```
减少小型 I/O
批量处理
控制刷新时机
管理缓冲区
```

# 13. C++ 中如何管理 `FILE *`

`FILE *` 是典型的资源句柄，必须执行：

```
fclose()
```

因此在 C++ 中适合使用 RAII。

例如：

```
#include <cstdio>
#include <memory>
#include <stdexcept>

using FilePtr = std::unique_ptr<FILE, decltype(&std::fclose)>;

FilePtr open_file(const char* path, const char* mode) {
    FILE* raw = std::fopen(path, mode);

    if (raw == nullptr) {
        throw std::runtime_error("fopen failed");
    }

    return FilePtr(raw, &std::fclose);
}
```

使用：

```
auto file = open_file("config.txt", "r");

char line[1024];

while (std::fgets(line, sizeof(line), file.get()) != nullptr) {
    // 处理数据
}
```

离开作用域后自动调用：

```
fclose()
```

不过需要注意：

> 如果业务非常关心关闭阶段的写入错误，不能完全忽略 `fclose()` 的返回值。

因为 `unique_ptr` 的删除器无法自然地把关闭错误返回给调用者。

这类场景可以设计显式的：

```
close()
```

或者专用文件类，在关键持久化路径中主动检查关闭和同步结果。

# 14. 常见错误和工程规则

## 14.1 不要使用 `while (!feof(fp))`

错误：

```
while (!feof(fp)) {
    fgets(line, sizeof(line), fp);
    process(line);
}
```

正确：

```
while (fgets(line, sizeof(line), fp) != NULL) {
    process(line);
}

if (ferror(fp)) {
    // 处理错误
}
```

## 14.2 不要使用 `fflush(stdin)` 清空输入

标准 C 中，`fflush()` 主要用于输出流。

```
fflush(stdin);
```

不是可移植的清空输入方式。

如果要丢弃当前输入行，应明确读取并消费剩余字符。

## 14.3 不要认为 `fflush()` 等于落盘

```
fflush → 用户态缓冲区进入内核
fsync  → 请求内核同步到底层设备
```

## 14.4 不要忽略短读和短写

`fread()` 返回值可能小于请求值。

可能原因包括：

- 到达 EOF
- 发生错误
- 只读取到部分元素

必须检查返回值。

## 14.5 不要默认 `fgets()` 一次得到完整一行

需要检查返回数据中是否包含：

```
'\n'
```

特别是处理外部输入时，应限制最大行长度，防止：

- 内存无限增长
- 恶意超长行
- 协议解析异常

## 14.6 不要无规则混用 `FILE *` 与 `fd`

混用时容易破坏：

- 缓冲状态
- 文件偏移量
- 所有权关系
- 关闭顺序

## 14.7 检查 `fclose()` 和 `fflush()` 的错误

尤其是写文件时：

```
if (fflush(fp) == EOF) {
    perror("fflush");
}

if (fclose(fp) == EOF) {
    perror("fclose");
}
```

不能因为前面的 `fprintf()` 返回成功，就认为数据最终一定写入成功。

## 14.8 明确谁负责关闭底层文件描述符

例如：

```
int fd = open(...);
FILE *fp = fdopen(fd, "r");
```

一旦 `fdopen()` 成功，通常由：

```
fclose(fp);
```

负责关闭 `fd`。

不要再额外执行：

```
close(fd);
```

否则可能出现重复关闭。

重复关闭的危险不仅是返回错误：如果文件描述符编号已经被系统复用，错误的第二次 `close()` 甚至可能关闭另一个完全无关的资源。

# 15. 用一句话理解常见函数

| 函数        | 工程理解                             |
| ----------- | ------------------------------------ |
| `fopen()`   | 打开文件并构造一个带缓冲的流对象     |
| `fdopen()`  | 给已有文件描述符套上标准 I/O 流      |
| `fileno()`  | 从标准 I/O 流取出底层文件描述符      |
| `fgets()`   | 从流中读取一段文本，通常以一行为目标 |
| `fread()`   | 从流中按块读取数据                   |
| `fprintf()` | 格式化后写入流的缓冲区               |
| `fwrite()`  | 向流中按块写入数据                   |
| `fflush()`  | 把用户态输出缓冲区提交给内核         |
| `feof()`    | 判断之前的读取是否已经遇到文件末尾   |
| `ferror()`  | 判断流是否发生错误                   |
| `fseek()`   | 同步流状态并修改逻辑文件位置         |
| `fclose()`  | 刷新数据、关闭 fd 并销毁流对象       |

# 16. 第五章真正应该掌握的知识

学习完第五章，不需要背诵全部 API，但应该掌握以下知识模型。

## 16.1 `FILE *` 是一个有状态的用户态对象

它不仅保存文件描述符，还维护：

- 缓冲区
- 当前读写位置
- EOF 标志
- 错误标志
- 缓冲策略

## 16.2 标准 I/O 的核心价值是批量处理

```
多次小型业务读写
        ↓
FILE 用户态缓冲区合并
        ↓
少量大型 read/write 系统调用
```

## 16.3 读取的核心路径

```
文件、终端、管道
        ↓
内核文件描述符
        ↓ read()
FILE 用户态缓冲区
        ↓ fgets/fread/fgetc
业务代码
```

## 16.4 写入的核心路径

```
业务代码
        ↓ fprintf/fwrite/fputs
FILE 用户态缓冲区
        ↓ write()
内核文件描述符
        ↓
文件、终端、管道
```

## 16.5 缓冲提高性能，也增加了状态复杂度

缓冲带来的收益是：

- 减少系统调用
- 提高吞吐量
- 提供高级读写接口

同时也带来了：

- 数据可能尚未真正写出
- 底层文件偏移量不直观
- `FILE *` 与 `fd` 不宜随意混用
- 错误可能延迟到刷新阶段出现
- 读写方向切换需要同步状态

## 16.6 `fflush()` 不等于 `fsync()`

```
业务代码
   ↓
FILE 用户态缓冲区
   ↓ fflush()
内核页缓存
   ↓ fsync()
持久化设备
```

# 17. 最终总结

第五章可以归纳为一句话：

> 标准 I/O 库通过 `FILE *` 在文件描述符之上维护用户态缓冲区和流状态，把底层的字节读写封装成按字符、按行、按块和按格式的数据处理接口。

完整结构为：

```
业务代码
   │
   │ 调用 fgets、fread、fprintf、fwrite
   ▼
FILE * 标准 I/O 流
   │
   │ 维护 fd、缓冲区、文件位置、EOF 和错误状态
   ▼
用户态缓冲区
   │
   │ 聚合小型读写，减少系统调用
   ▼
read / write
   │
   │ 从用户态进入内核态
   ▼
内核文件描述符
   │
   │ 引用内核中的打开文件对象
   ▼
文件、终端、管道、设备
```

对于 C++ 服务端工程师，学习本章的重点不是记住所有函数，而是理解：

1. `FILE *` 与文件描述符的关系
2. 用户态缓冲区为什么能够提高性能
3. 一次标准 I/O 调用不等于一次系统调用
4. `fflush()`、`fsync()` 和 `fclose()` 分别处于什么层次
5. 为什么不能随意混用标准 I/O 和文件描述符 I/O
6. 为什么高并发网络服务通常要自己管理连接缓冲区
7. 如何通过 RAII 管理 `FILE *` 的生命周期

只要这些问题能够清楚回答，第五章的核心内容就已经真正掌握。
