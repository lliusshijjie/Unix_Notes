# 《UNIX环境高级编程》第 7 章：进程环境

---

## 1. 本章定位

这一章并不重点讲“如何创建进程”，而是先回答：

> 一个程序被装载成进程以后，在进入 `main()` 前、运行期间以及退出时，究竟处于怎样的环境中？

本章建立的是 **Unix 进程运行时模型**，主要包括：

- 程序如何进入 `main()`；
- 进程如何正常或异常终止；
- 命令行参数和环境变量如何传入进程；
- 进程虚拟地址空间由哪些区域组成；
- `malloc` 背后的内存分配机制；
- 如何进行跨函数层级的非局部跳转；
- 操作系统如何限制进程可使用的资源。

它是后续学习进程控制、信号、线程和进程间通信的基础章节。

# 3. 各小节知识总结

## 3.1 `main` 函数

常见形式：

```cpp
int main(int argc, char* argv[]) {
    return 0;
}
```

需要理解的重点：

- `main()` 是用户程序的入口，但不是进程真正的第一条指令；
- 在 `main()` 之前，运行库启动代码已经完成了一系列初始化；
- 启动代码会接收内核准备的命令行参数和环境变量；
- `main()` 返回后，运行库通常会调用 `exit()` 完成正常终止。

大致流程：

```text
内核装载程序
    ↓
运行库启动代码
    ↓
初始化运行环境
    ↓
调用 main(argc, argv)
    ↓
main 返回
    ↓
exit
```

对 C++ 程序而言，`main()` 前后还可能包含：

- 全局对象构造；
- 静态对象初始化；
- 全局对象析构；
- 运行库清理。

### 工程认识

不要把进程生命周期简单理解成：

```text
进入 main → 离开 main
```

更准确的是：

```text
装载 → 运行库初始化 → main → 运行库清理 → 内核回收
```

---

## 3.2 进程终止

进程终止分为两大类：

### 正常终止

常见方式：

- 从 `main()` 返回；
- 调用 `exit()`；
- 调用 `_exit()` 或 `_Exit()`；
- 进程的最后一个线程正常返回；
- 最后一个线程调用线程退出函数。

### 异常终止

常见方式：

- 调用 `abort()`；
- 收到导致终止的信号；
- 最后一个线程响应取消请求。

本章最重要的区分是：

```text
exit()
    ↓
执行用户态清理
    ├── 调用终止处理函数
    ├── 刷新标准 I/O 缓冲区
    └── 关闭标准 I/O 流
    ↓
进入内核终止

_exit() / _Exit()
    ↓
不执行完整的标准 I/O 清理
    ↓
直接进入内核终止
```

### `return` 与 `exit`

从 `main()` 返回：

```cpp
return 0;
```

通常等价于：

```cpp
exit(0);
```

但是在普通函数中执行 `return`，只会返回上层调用者，不会终止整个进程。

### 终止状态

进程会向父进程提供一个终止状态：

```cpp
return 0;
exit(EXIT_SUCCESS);
exit(EXIT_FAILURE);
```

后续学习 `wait()` 和 `waitpid()` 时，父进程会读取这个状态。

### 工程认识

服务端程序退出前通常需要：

- 停止接收新请求；
- 等待正在执行的请求结束；
- 刷新日志；
- 关闭线程池；
- 关闭数据库和网络连接；
- 保存必要状态。

但不能假设清理代码一定执行，例如：

- 进程收到 `SIGKILL`；
- 机器断电；
- 操作系统崩溃。

关键数据不能只依赖“进程退出时保存”。

## 3.3 终止处理函数 `atexit`

`atexit()` 可以登记进程正常终止时需要执行的函数：

```cpp
#include <cstdlib>

void cleanup() {
    // 进程级清理
}

int main() {
    std::atexit(cleanup);
}
```

多个终止处理函数通常按照：

```text
后注册，先执行
```

即类似栈的顺序。

### 适用场景

- 清理进程级别的全局资源；
- 兼容 C 风格库；
- 执行简单的运行时收尾工作。

### C++ 工程建议

普通资源管理应优先使用 RAII：

```cpp
{
    File file("data.txt");
    // 离开作用域时自动释放
}
```

RAII 管理的是对象和作用域生命周期；`atexit()` 管理的是整个进程的正常退出阶段。

不要把所有资源清理都集中到 `atexit()`。

## 3.4 命令行参数

命令行参数通过 `argc` 和 `argv` 传入：

```cpp
int main(int argc, char* argv[]) {
    for (int i = 0; i < argc; ++i) {
        // argv[i]
    }
}
```

例如：

```bash
./server --port 8080 --config server.toml
```

对应：

```text
argv[0] = "./server"
argv[1] = "--port"
argv[2] = "8080"
argv[3] = "--config"
argv[4] = "server.toml"
```

通常：

```text
argv[argc] == nullptr
```

### 常见用途

- 指定配置文件；
- 指定监听地址和端口；
- 设置日志等级；
- 切换前台或后台运行模式；
- 开启调试功能。

### 工程建议

不要在业务代码中到处直接读取 `argv`。更合理的方式是：

```text
argv
  ↓
参数解析层
  ↓
Config 配置对象
  ↓
业务模块
```

## 3.5 环境表

每个进程都有一张环境表，本质上是一组字符串：

```text
KEY=value
```

例如：

```text
PATH=/usr/local/bin:/usr/bin
HOME=/home/user
LANG=zh_CN.UTF-8
LOG_LEVEL=info
```

环境表通常可以通过全局变量访问：

```cpp
extern char** environ;
```

其结构可以理解为：

```text
environ
   │
   ├── "PATH=..."
   ├── "HOME=..."
   ├── "LANG=..."
   └── nullptr
```

### 关键认识

环境变量不是整个系统共享的一份全局变量，而是：

> 每个进程都拥有自己的环境表。

子进程通常从父进程继承一份环境，但之后可以独立修改。

```text
父进程环境表
      │
      │ 创建并执行子进程
      ▼
子进程获得环境表
```

这为后续学习 `fork()` 和 `exec()` 的继承关系打基础。

## 3.6 C 程序的存储空间布局

典型进程虚拟地址空间可以简化为：

```text
高地址
┌────────────────────────────┐
│ 栈                          │
│ 函数调用、局部变量          │
│ 通常向低地址增长            │
├────────────────────────────┤
│ 内存映射区                  │
│ 共享库、mmap 文件、匿名映射 │
├────────────────────────────┤
│ 堆                          │
│ malloc / new                │
│ 通常向高地址增长            │
├────────────────────────────┤
│ BSS                         │
│ 未显式初始化的全局/静态变量 │
├────────────────────────────┤
│ 初始化数据段                │
│ 已初始化的全局/静态变量     │
├────────────────────────────┤
│ 正文段                      │
│ 程序机器指令、只读数据      │
└────────────────────────────┘
低地址
```

### 正文段

保存程序机器指令，通常具有：

```text
可读 + 可执行
```

多个进程运行同一个程序时，正文段可能由系统共享。

### 初始化数据段

保存具有明确非零初始值的全局变量和静态变量：

```cpp
int global_count = 10;
static int cache_size = 1024;
```

### BSS 段

保存未显式初始化或初始化为零的全局、静态变量：

```cpp
int global_count;
static int cache_size = 0;
```

可执行文件通常只记录 BSS 所需空间大小，不需要真正保存大量零。

### 堆

用于动态内存分配：

```cpp
auto ptr = std::make_unique<int[]>(1024);
```

### 栈

用于保存：

- 局部变量；
- 函数参数；
- 返回地址；
- 寄存器现场；
- 函数调用上下文。

在多线程程序中，每个线程通常都有自己的栈，而堆和全局数据由进程内线程共享。

### 注意

该图是理解模型，不代表所有 Unix 系统和所有架构都完全采用相同的地址排列。现代系统还会受到：

- ASLR；
- 动态链接；
- 线程栈；
- `mmap`；
- 内存保护机制

等因素影响。

## 3.7 共享库

共享库允许多个程序复用相同的库代码。

静态链接的大致关系：

```text
程序目标文件 + 静态库
          ↓
      可执行文件
```

共享库链接的大致关系：

```text
可执行文件
    │
    └── 记录依赖的共享库
              │
              ▼
       程序运行时装载
```

### 共享库的主要价值

- 减小可执行文件体积；
- 多个进程可以共享只读代码页；
- 更新库时不一定需要重新链接所有程序；
- 便于模块复用。

### 共享库带来的问题

- 运行时依赖缺失；
- 版本兼容；
- ABI 不兼容；
- 动态库搜索路径错误；
- 符号冲突。

后续排查 Linux 服务启动失败时，经常会用到：

```bash
ldd ./server
```

共享库知识还会与以下内容发生联系：

- ELF 文件；
- 动态链接器；
- 地址空间映射；
- 符号解析；
- 插件系统。

## 3.8 存储空间分配

C 语言提供的基础动态内存接口包括：

```c
malloc
calloc
realloc
free
```

### `malloc`

分配指定字节数，内容未初始化：

```cpp
void* p = std::malloc(size);
```

### `calloc`

按“元素数量 × 单个元素大小”分配，并将空间初始化为零：

```cpp
void* p = std::calloc(count, size);
```

### `realloc`

调整已有内存块大小：

```cpp
p = std::realloc(p, new_size);
```

原有数据会在一定范围内保留，但内存块可能移动到新地址。

### `free`

释放动态内存：

```cpp
std::free(p);
```

### 一个重要误区

不要理解为：

```text
每次 malloc
    ↓
立即执行一次系统调用
```

更接近真实情况的是：

```text
业务代码
    ↓
malloc / new
    ↓
用户态内存分配器
    ├── 从已有空闲块中分配
    └── 空间不足时，通过 brk 或 mmap 向内核申请
```

内存分配器会维护用户态内存池，以减少系统调用。

### 常见问题

- 内存泄漏；
- 重复释放；
- 释放后继续访问；
- 越界写；
- 内存碎片；
- `realloc` 失败后丢失原指针；
- 多线程竞争和分配器锁开销。

### C++ 工程建议

优先使用：

```cpp
std::vector
std::string
std::unique_ptr
std::shared_ptr
std::make_unique
std::make_shared
```

只有在底层库、兼容 C 接口、定制内存管理等场景中，才直接使用 `malloc/free`。

## 3.9 环境变量操作

读取环境变量：

```cpp
const char* value = std::getenv("HOME");
```

常见操作接口包括：

```text
getenv
putenv
setenv
unsetenv
```

### `getenv`

查询环境变量：

```cpp
const char* value = std::getenv("LOG_LEVEL");
```

返回值可能为空，使用前要检查。

### `setenv`

设置或覆盖环境变量：

```c
setenv("LOG_LEVEL", "debug", 1);
```

### `unsetenv`

删除环境变量：

```c
unsetenv("LOG_LEVEL");
```

### `putenv`

把给定字符串放入环境表。其字符串生命周期和所有权语义容易引发问题，工程中通常优先考虑 `setenv()`。

### 服务端应用

环境变量经常用于：

- 区分开发、测试、生产环境；
- 容器配置注入；
- 日志级别；
- 服务地址；
- 凭据路径；
- 动态库搜索路径。

### 工程建议

环境变量适合做外部配置输入，但应在进程启动阶段集中读取并校验：

```text
环境变量
   ↓
Config 加载与校验
   ↓
生成不可变配置对象
   ↓
业务模块使用
```

不要让业务模块随时读取和修改全局环境。

## 3.10 `setjmp` 与 `longjmp`

`setjmp()` 和 `longjmp()` 提供非局部跳转能力。

普通函数调用返回：

```text
main
  ↓
func_a
  ↓
func_b
  ↓
func_c
  ↑
逐层 return
```

非局部跳转：

```text
main 中 setjmp
  ↓
func_a
  ↓
func_b
  ↓
func_c 中 longjmp
  └──────────────→ 直接跳回 setjmp
```

示意代码：

```c
jmp_buf env;

if (setjmp(env) == 0) {
    deep_function();
} else {
    /* 从 longjmp 返回 */
}
```

深层函数：

```c
longjmp(env, 1);
```

### 它解决的问题

在没有语言级异常机制的 C 程序中，可以从深层调用栈直接返回到预先设置的位置。

### 风险

- 控制流难以理解；
- 可能绕过正常的资源释放；
- 自动变量的值需要谨慎处理；
- 容易破坏程序状态。

### C++ 中尤其危险

`longjmp()` 可能绕过 C++ 对象的析构和栈展开：

```cpp
void work() {
    std::lock_guard<std::mutex> lock(mutex);
    longjmp(env, 1);
}
```

可能导致锁没有正常释放。

### 工程结论

对现代 C++ 服务端开发：

> 理解机制即可，不应把 `setjmp/longjmp` 当作普通错误处理方案。

优先考虑：

- 返回错误码；
- `std::expected`；
- 异常；
- RAII；
- 显式状态机。

## 3.11 进程资源限制

Unix 为每个进程设置资源上限，可通过：

```c
getrlimit
setrlimit
```

查询或修改。

每类资源通常包含：

```text
软限制：进程当前实际使用的限制
硬限制：软限制能够提升到的最大值
```

普通进程通常可以降低限制，也可以在硬限制范围内调整软限制，但不能随意提高硬限制。

### 常见资源限制

| 资源 | 含义 | 服务端常见影响 |
|---|---|---|
| `RLIMIT_NOFILE` | 最大文件描述符数量 | 高并发连接、日志文件、管道 |
| `RLIMIT_CORE` | core dump 最大大小 | 崩溃后能否生成完整 core |
| `RLIMIT_STACK` | 线程栈大小限制 | 深递归、大局部数组、线程数量 |
| `RLIMIT_AS` | 进程虚拟地址空间限制 | 内存申请失败 |
| `RLIMIT_CPU` | CPU 时间限制 | 超限后收到信号 |
| `RLIMIT_NPROC` | 用户可创建的进程/线程数量 | 创建线程或子进程失败 |

查看 shell 的资源限制：

```bash
ulimit -a
```

查看运行中进程的限制：

```bash
cat /proc/<pid>/limits
```

### 服务端高频故障

文件描述符不足：

```text
accept 失败
socket 创建失败
open 失败
日志文件打开失败
```

常见错误：

```text
Too many open files
```

线程或进程数量受限：

```text
pthread_create 失败
fork 失败
```

core 大小受限：

```text
程序崩溃
    ↓
没有生成 core 文件
    ↓
难以进行事后调试
```

### 工程结论

资源限制属于部署环境的一部分。程序在生产环境中启动时，应检查关键限制是否符合预期，而不是只关注业务配置。

# 5. 与前面章节的联系

## 5.1 与文件 I/O 的联系

Unix 进程运行期间持有文件描述符：

```text
进程
  └── 文件描述符表
        ├── 普通文件
        ├── socket
        ├── 管道
        └── 终端
```

进程退出后，内核会关闭其仍然打开的文件描述符。

后续学习 `fork()` 时，还会研究文件描述符如何被子进程继承。

---

## 5.2 与标准 I/O 的联系

`exit()` 与 `_exit()` 的区别，必须结合标准 I/O 缓冲区理解：

```text
FILE*
  ↓
用户态缓冲区
  ↓
write
  ↓
内核文件描述符
```

`exit()` 会执行标准 I/O 清理并刷新缓冲区；`_exit()` 不会执行完整的标准 I/O 刷新流程。

这会直接影响后续的 `fork()` 行为。

# 6. 与未来章节的联系

## 6.1 与第 8 章“进程控制”的联系

第 8 章会学习：

- `fork()`；
- `vfork()`；
- `exec` 函数族；
- `wait()`；
- `waitpid()`；
- 僵尸进程；
- 进程终止状态。

第 7 章提供了理解这些内容的前提。

### 地址空间与 `fork`

必须先知道进程地址空间由什么组成，才能理解：

```text
父进程 fork
    ↓
子进程获得父进程地址空间的逻辑副本
    ↓
通常通过写时复制优化
```

### 环境变量与 `exec`

必须先理解环境表，才能理解：

```text
exec
  ├── 使用当前环境
  └── 显式传入新环境
```

### `exit` 与 `_exit`

子进程在 `fork()` 后、`exec()` 前发生错误时，经常使用 `_exit()`，避免：

- 重复刷新父进程复制过来的标准 I/O 缓冲区；
- 执行父进程注册的退出处理函数。

### 终止状态与 `wait`

本章讲进程产生终止状态，下一章讲父进程如何获取终止状态。

```text
子进程退出
    ↓
保存终止信息
    ↓
父进程 wait/waitpid
    ↓
回收子进程
```

## 6.3 与第 10 章“信号”的联系

信号可能导致进程异常终止：

```text
信号到达
   ↓
执行默认动作、忽略或调用处理函数
   ↓
进程可能终止
```

本章对正常终止和异常终止的划分，是学习信号终止语义的基础。

另外，`setjmp/longjmp` 的思想还会与信号处理中的跳转恢复产生联系，但实际使用需要高度谨慎。

## 6.4 与线程章节的联系

## 6.5 与进程间通信章节的联系

进程通常拥有独立的虚拟地址空间，因此不同进程不能直接访问彼此的普通变量。

这解释了为什么需要：

- 管道；
- FIFO；
- 消息队列；
- 共享内存；
- socket；
- 信号量。

```text
进程 A 独立地址空间
        │
        │ IPC 机制
        ▼
进程 B 独立地址空间
```

共享内存是一个特殊情况：多个进程把同一内核对象映射到各自的地址空间中。

## 6.6 与守护进程和服务端程序的联系

服务端程序常见启动流程：

```text
读取命令行参数
    ↓
读取环境变量
    ↓
检查资源限制
    ↓
初始化日志和共享库
    ↓
创建线程或子进程
    ↓
进入事件循环
```

退出流程：

```text
收到终止信号
    ↓
停止接受新请求
    ↓
等待存量请求结束
    ↓
释放业务资源
    ↓
正常退出
```

因此，第 7 章看似偏基础，实际上贯穿服务器整个生命周期。

# 8. 常见误区

## 误区一：程序从 `main()` 的第一行开始执行

实际上：

```text
内核装载
  ↓
运行库初始化
  ↓
main
```

---

## 误区二：每次 `malloc` 都会执行系统调用

实际上内存分配器通常先从用户态内存池中分配。

---

## 误区三：环境变量是系统中的全局变量

实际上每个进程都有自己的环境表，子进程通常继承父进程环境。

---

## 误区四：`exit()` 和 `_exit()` 没有区别

`exit()` 会执行用户态清理，`_exit()` 更直接地结束进程。后续 `fork()` 场景中区别非常关键。

---

## 误区五：进程退出后内核会回收资源，因此无需清理

内核可以回收内核资源和地址空间，但无法替代业务层面的：

- 日志刷新；
- 状态提交；
- 临时文件处理；
- 优雅关闭；
- 下游通知。

---

## 误区六：栈和堆就是进程全部内存

进程地址空间还包括：

- 正文段；
- 初始化数据段；
- BSS；
- 共享库；
- 文件映射；
- 匿名映射；
- 线程栈。
